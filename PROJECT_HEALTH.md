## 技术债务

### 高优先级

| 债务 | 影响 | 偿还建议 |
|------|------|----------|
| 单文件架构 | 新增功能会使 main.rs 持续膨胀 | 拆分为 video/audio/input/emulator 模块 |
| `let _ =` 错误丢弃 | SDL2 调用失败时无法感知 | 替换为 `if let Err(e)` 或 `?` 传播 |
| 无配置系统 | 键位、窗口、音频参数全部硬编码 | 引入 config.toml + 运行时配置结构 |

### 中优先级

| 债务 | 影响 | 偿还建议 |
|------|------|----------|
| 无单元测试 | 重构风险高，回归难检测 | 为输入映射、字体渲染等添加测试 |
| 音频缓冲区无上限 | 长时间运行可能内存增长 | 添加缓冲区大小限制或环形缓冲 |
| `block v0.1.6` 警告 | 未来 Rust 版本可能编译失败 | 更新依赖或等待上游修复 |

### 低优先级

| 债务 | 影响 | 偿还建议 |
|------|------|----------|
| 硬编码窗口标题 | 无法动态显示 ROM 名称 | 从 iNES header 提取名称 |
| 硬编码字体 | 字符集有限，无法扩展 | 迁移到 SDL2_ttf 或嵌入完整字体 |

## 风险点

### 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| tetanes-core API 变更 | 中 | 编译失败 | 锁定版本 0.14，升级前充分测试 |
| SDL2 macOS 兼容性 | 低 | 渲染/输入异常 | 多版本测试 |
| Rust 未来版本兼容 | 低 | 编译失败 | 关注 `block` crate 更新 |
| 音频缓冲区溢出 | 低 | 内存增长 | 监控长时间运行表现 |

### 项目风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 功能蔓延 | 中 | 代码质量下降 | 严格遵循阶段开发，每个阶段可编译运行 |
| 单文件膨胀 | 高 | 可维护性下降 | 适时拆分模块，保持 main.rs < 800 行 |
| 缺少测试 | 高 | 重构困难 | 优先为核心逻辑添加测试 |

## 推荐开发顺序

基于依赖关系和风险收益比：

```
Phase 1: 基础稳定性（当前）
  ├── [x] Controls Overlay
  ├── [x] L1 输入修复
  └── [x] v0.1 基线建立

Phase 2: 配置系统（下一步推荐）
  ├── config.toml 配置文件
  ├── 运行时配置结构
  └── 可配置键位映射
  理由：后续功能（Save State、菜单）都依赖配置系统

Phase 3: 核心体验
  ├── Save State
  ├── Quick Reset
  ├── 暂停功能
  └── 音量控制
  理由：提升游戏体验，用户直接感知

Phase 4: 界面增强
  ├── Fullscreen
  ├── FPS Overlay
  ├── Turbo 按钮
  └── 窗口缩放
  理由：锦上添花，优先级低于核心体验

Phase 5: 架构优化（按需）
  ├── 模块拆分
  ├── 单元测试
  └── 错误处理改进
  理由：随功能增长适时重构，避免过早抽象
```

### 关键原则

1. **每个阶段可编译运行** — 不引入不可运行的中间状态
2. **先功能后重构** — 功能驱动架构演进，不为重构而重构
3. **配置系统优先** — 它是后续功能的基础设施
4. **测试跟随** — 为核心逻辑添加测试，不追求覆盖率

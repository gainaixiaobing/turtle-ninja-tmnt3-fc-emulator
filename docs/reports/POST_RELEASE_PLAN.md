# TurtleBox Post-Release Plan

## 1. 发布后观察指标

### 崩溃

| 监控点 | 触发条件 | 排查方向 |
|--------|----------|----------|
| ROM 加载失败 | 用户打开损坏/不支持的 ROM | `deck.load_rom_path()` 错误处理 |
| 全屏切换崩溃 | macOS 特定分辨率/显示器配置 | `FullscreenType::Desktop` 兼容性 |
| 存档加载崩溃 | 损坏的 .state 文件 | `deck.load_state()` 反序列化失败 |
| 长时间运行 | 连续游玩 30+ 分钟 | 音频缓冲区无限增长、内存泄漏 |

### 音频问题

| 监控点 | 可能表现 | 排查方向 |
|--------|----------|----------|
| 无声音 | 启动后完全静音 | SDL2 音频设备初始化 |
| 爆音/杂音 | 间歇性噪音 | 音频缓冲区欠载/过载 |
| 音量不生效 | 按 =/- 无反应 | volume_factor 未正确应用 |
| 暂停后爆音 | 恢复瞬间有噪音 | 音频缓冲区未完全清空 |

### 输入问题

| 监控点 | 可能表现 | 排查方向 |
|--------|----------|----------|
| 手柄未识别 | 连接后无反应 | SDL2 GameController 枚举 |
| 按键映射错误 | 按 A 触发 B | config.toml 解析错误 |
| 配置修改无效 | 编辑 config.toml 后行为不变 | 大小写匹配问题 |
| 手柄断开重连 | 拔插后失效 | ControllerDeviceRemoved 处理 |

### Save State 问题

| 监控点 | 可能表现 | 排查方向 |
|--------|----------|----------|
| 存档文件未创建 | 按 F5 无文件 | `./saves/` 目录权限 |
| 加载后状态错误 | 画面/声音不一致 | tetanes-core 版本兼容性 |
| 跨版本加载 | 升级后旧存档失效 | SAVE_VERSION 不匹配 |
| 文件损坏 | 加载时 panic | 头部校验 + deflate 校验 |

## 2. 用户反馈分类规则

### Bug

定义：程序行为与预期不符，或导致崩溃、数据丢失。

示例：
- 按 F5 后程序崩溃
- 全屏模式下画面撕裂
- 手柄按键无响应
- 存档加载后游戏状态错误

处理优先级：P0（崩溃）> P1（功能失效）> P2（视觉/音频异常）

### UX

定义：功能正常但体验不佳，或用户预期与实际行为不一致。

示例：
- 音量调节步长太大/太小
- 暂停 overlay 遮挡游戏画面
- 全屏切换有延迟
- 存档路径不直观

处理优先级：根据影响范围评估

### Feature Request

定义：用户期望的新能力，当前版本未实现。

示例：
- 多存档槽位
- 截图功能
- 回放功能
- CRT 滤镜

处理优先级：归入 v0.3 候选，不立即开发

## 4. Release Retrospective

### v0.1 → v0.2 成功经验

**Sprint 模式有效。** 将开发拆分为 5 个 Sprint，每个 Sprint 聚焦单一目标，验证后才进入下一个。这避免了功能膨胀和架构漂移。

**模块化策略正确。** Sprint 1 提取 input.rs 和 overlay.rs，后续 Sprint 只修改 state.rs 和 main.rs 少量代码。main.rs 从 318 行增长到 337 行（+19），而 state.rs 独立承担 219 行新逻辑。

**配置系统前置。** Sprint 2 建立配置基础设施，Sprint 3 立即应用。配置驱动的设计让后续功能（全屏、音量）无需额外适配。

**文档与代码同步。** 每个 Sprint 完成后生成报告，审计时有据可查。v0.2 最终有 42 份 .md 文档覆盖全生命周期。

### 架构决策效果

| 决策 | 效果 | 评价 |
|------|------|------|
| 使用 tetanes-core | 无需实现 CPU/PPU/APU | 正确，节省大量工作 |
| state.rs 封装新功能 | main.rs 几乎不增长 | 正确，保持可读性 |
| 配置使用 TOML | 用户可手动编辑 | 正确，降低使用门槛 |
| SDL2 Desktop 全屏 | macOS 兼容性好 | 正确，避免全屏独占问题 |
| bincode + deflate 存档 | 文件小、速度快 | 正确，上游已验证 |

### 后续可复用流程

1. **Sprint 规划模式**：每个 Sprint 有明确目标、验证清单、产出报告
2. **模块化优先**：新功能优先封装在独立模块，避免 main.rs 膨胀
3. **配置驱动设计**：新功能尽量通过配置暴露，便于用户定制
4. **审计先行**：在编码前先做可行性研究和架构分析
5. **文档同步**：每个阶段产出报告，形成完整项目记录
6. **编译验证**：每个 Sprint 完成后必须 cargo fmt + build + clippy
7. **功能冻结**：Release 阶段严格禁止代码修改，只做审查

### 1.3 数据流架构

```
ROM 文件
  ↓ deck.load_rom_path()
ControlDeck
  ↓ deck.clock_frame()
  ├── deck.frame_buffer() → SDL2 Texture → canvas.copy() → canvas.present()
  ├── deck.audio_samples() → Arc<Mutex<Vec<f32>>> → SDL2 AudioCallback
  └── deck.joypad_mut() ← SDL2 Event Pump (Keyboard + Controller)
```

**审计结论：** 数据流清晰，单向管道，无循环依赖。`ControlDeck` 是唯一的状态持有者。

## 2. 技术债务审计

### 2.1 错误处理

| 位置 | 问题 | 严重度 |
|------|------|--------|
| `main.rs:25` | `self.buffer.lock().unwrap()` — Mutex 中毒时 panic | **低** |
| `main.rs:126` | `let _ = canvas.fill_rect(...)` — 静默丢弃 SDL 绘制错误 | **中** |
| `main.rs:151` | `let _ = canvas.fill_rect(...)` — 同上 | **中** |
| `main.rs:335` | `Err(_) => {}` — 手柄打开失败静默忽略 | **低** |
| `main.rs:506` | `audio_buffer.lock().unwrap()` — Mutex 中毒时 panic | **低** |

**严重度说明：**
- `let _ =` 模式在渲染路径中出现 2 次。如果 SDL2 渲染器进入错误状态，所有绘制调用静默失败，用户看到黑屏但无错误提示。
- `unwrap()` 在音频回调中出现 2 次。如果 Mutex 中毒（任何线程 panic 时），音频回调将 panic，导致整个进程崩溃。

### 2.2 硬编码值

| 位置 | 硬编码值 | 影响 |
|------|----------|------|
| `main.rs:14` | `STICK_THRESHOLD: i16 = 16000` | 不可配置 |
| `main.rs:267` | 窗口大小 `960, 720` | 不可调整 |
| `main.rs:281` | 纹理大小 `256, 240` | NES 固有，合理 |
| `main.rs:287-289` | 音频 `44100 Hz, 1 ch, 1024 samples` | 不可配置 |
| `main.rs:349` | 帧间隔 `16_639 μs` | NTSC 固有，合理 |
| `main.rs:153` | overlay scale `3` | 不可配置 |
| `main.rs:155` | overlay margin `50` | 不可配置 |

**严重度：低** — 对 v0.1 可接受，但配置系统应列为 v0.2 首要任务。

### 2.3 重复代码

| 位置 | 重复模式 | 严重度 |
|------|----------|--------|
| `main.rs:364-391` | KeyDown/KeyUp 的 match arm 完全对称（8 对映射） | **低** |
| `main.rs:413-456` | ControllerButtonDown/ButtonUp 的 match arm 完全对称 | **低** |

**影响：** 新增按键时需同步修改两处。当前规模可接受，超过 12 对映射时应提取为查表。

### 3.2 模块化程度

当前：**零模块拆分**。所有逻辑在 `main.rs`。

最小可行拆分方案：
```
src/
├── main.rs        — 入口 + 主循环
├── audio.rs       — NesAudioCallback + 音频管道
├── overlay.rs     — 字体 + overlay 渲染
├── input.rs       — 输入映射表
└── rom.rs         — ROM 路径解析 + 验证
```

**严重度：中** — 不紧急，但 v0.2 如新增 Save State 或配置系统，`main()` 将超过 400 行。

## 4. 性能审计

### 4.1 主循环性能

| 阶段 | 代码行 | 性能特征 |
|------|--------|----------|
| 事件处理 | L354-492 | O(events_per_frame)，通常 < 10 事件 |
| clock_frame() | L495 | **瓶颈** — NES 模拟，~29780 CPU cycles/frame |
| 音频复制 | L504-508 | O(audio_samples)，~3607 samples/frame |
| 纹理更新 | L515 | O(256×240×4) = 245,760 bytes，DMA 传输 |
| canvas.copy() | L520 | GPU 缩放，可忽略 |
| Overlay 渲染 | L523-525 | O(characters × 35)，仅 F1 时触发 |
| 帧率限制 | L530-533 | `thread::sleep`，精度依赖 OS |

**审计结论：** 主循环无明显性能瓶颈。`clock_frame()` 是计算密集型，但 NES 模拟在现代 CPU 上远低于 16.6ms 帧预算。

### 4.2 音频管道

```rust
// L506-507
let mut buffer = audio_buffer.lock().unwrap();
buffer.extend_from_slice(audio_samples);
```

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 缓冲区无上限 | **低** | `Vec<f32>` 持续增长，但 drain 在回调中消耗。正常运行下大小稳定在 ~3607 samples。异常情况（音频设备卡住）可能导致内存增长 |
| Mutex 竞争 | **低** | 主循环每帧 lock 一次，回调每 1024 samples lock 一次。竞争概率低 |

### 4.3 Overlay 渲染性能

```rust
// L119-134 draw_char()
for row in 0..7 {
    for col in 0..5 {
        if bit_set { canvas.fill_rect(...) }  // 最多 35 次 fill_rect/字符
    }
}
```

| 问题 | 严重度 | 说明 |
|------|--------|------|
| fill_rect 调用次数 | **低** | 19 行文本 × ~15 字符 × ~18 像素/字符 ≈ 5130 次 fill_rect。仅在 F1 时触发，不影响正常游戏 |
| 每帧设置 blend_mode | **低** | `set_blend_mode` 每次 overlay 调用时设置，应缓存 |

### 4.4 帧率限制精度

```rust
// L530-533
let elapsed = frame_start.elapsed();
if elapsed < frame_duration {
    std::thread::sleep(frame_duration - elapsed);
}
```

| 问题 | 严重度 | 说明 |
|------|--------|------|
| `thread::sleep` 精度 | **中** | macOS 上 `sleep` 精度约 1-10ms，可能导致帧时间抖动。目标 16.639ms，实际可能 15-20ms 波动 |
| 无 vsync | **低** | 未使用 `PresentVSync`，依赖 sleep 限制帧率。可能产生画面撕裂 |
